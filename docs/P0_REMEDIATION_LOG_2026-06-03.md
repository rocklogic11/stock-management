# P0 整改日志 - 2026-06-03

## 背景

根据《库存预备齐_专家评审报告》及项目复核，本次 P0 聚焦会直接影响账实一致性、生产安全和敏感数据暴露的问题。UI 体验、PWA 离线能力、CI/CD 完整流水线等纳入 P1/P2 后续迭代。

## 本次整改范围

1. 盘点审核库存快照一致性
   - 盘点单创建时继续记录 `system_quantity` 作为快照。
   - 审核通过前比对商品当前库存与盘点快照。
   - 审核事务中显式锁定相关商品行，降低并发入库/盘点审核同时发生造成竞态覆盖的风险。
   - 如果盘点期间库存已变化，接口返回 `409 / code=1006`，并返回冲突商品清单，阻止直接覆盖库存。
   - 审核通过后按盘点快照到实盘数量生成库存流水，避免流水前后数量与盘点依据不一致。

2. 入库创建与确认两阶段
   - `POST /api/v1/inbound-orders` 不再接受前端传入的 `status` 直接完成入库。
   - 新建入库单固定为草稿/待确认状态 `status=2`，不改库存。
   - 只有 `PUT /api/v1/inbound-orders/:id/confirm` 才执行库存增加、加权成本更新和库存流水记录。
   - 前端“确认入库”流程调整为先创建草稿，再调用确认接口。

3. JWT 与生产环境变量加固
   - 生产环境启动时强制校验 `JWT_SECRET`、`JWT_REFRESH_SECRET`、数据库连接等关键环境变量。
   - 生产环境禁止使用默认 JWT secret。
   - 新增 `server/.env.example`、`web/.env.example` 作为模板。
   - 将 `server/.env.production`、`web/.env.production` 从 Git 跟踪中移除，避免继续提交生产凭据。

4. 敏感成本字段服务端过滤
   - 新增统一权限工具 `server/src/utils/permissions.js`。
   - 商品、入库单、盘点单接口统一基于服务端角色/权限过滤成本价、入库单金额、盘点差异金额等字段。
   - 店员即使绕过前端页面直接调接口，也不应看到成本相关字段。

5. 自动化验证
   - 新增 `server/src/scripts/p0-regression-test.js`。
   - `server/package.json` 增加 `npm run test:p0`。
   - `scripts/release-check.ps1` 增加 P0 回归测试步骤。
